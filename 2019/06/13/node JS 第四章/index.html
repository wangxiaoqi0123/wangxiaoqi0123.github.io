身份验证

  • 原因
    • http请求是无状态的，不知道访问者是谁
    • 频繁登录，增大服务器开销
  • 方案
    • session+cookie
      • 区别
        • cookie存放在浏览器上，session存放在服务器上
        • 单个cookie保存的数据不能超过4K，session理论上无限制
        • cookie不是很安全
        • 不允许JS访问跨域的Cookie
      • 联系
        • 都是用来记录客户端的信息
        • session的id值保存在cookie中
      • 使用
        • express-session
        • 前端登录成功，后端返回session
        • cookie不需要手动操控
        • 后端拦截路由，获取session，并作判断
    • jwt
      • 特点
        • 减少不必要的上传，减轻服务端压力
        • 避免受到CSRF跨站伪造的请求攻击
        • json的通用性，支持跨语言请求
      • 使用
        • jwt
          • sign
          • verify
        • 封装
          • createToken
          • verifyToken
        • 登录成功，后端返回token
        • 前端获取token，存到localStorage
        • 前端从localStorage读取token，上传后端
        • 后端拦截路由，获取token，并校验

cookie、localstorage等之间的区别

  • cookie数据始终在同源（需要指定作用域，不可以跨域调用）的http请求中携带（即使不需要），即在浏览器和服务器间来回传递，随着HTTP事务一起被发送，会浪费一部分带宽；对于sessionStorage和localStorage，一旦数据保存在本地后，就可以避免再向服务器请求数据，减少数据在浏览器和服务器间不必要地来回传递。
  • cookie中每条cookie的存储空间为4k，localStorage中一般浏览器支持的是5M。
  • cookie只在设置的过期时间之前一直有效，即使窗口或浏览器关闭；localStorage用作持久保存数据，将数据保存在客户端本地的硬件设备中；而很多时候数据只需要在用户浏览一组页面期间使用，关闭窗口后数据就可以丢弃了，这种情况使用sessionStorage非常方便，它将数据保存在session对象中。
  • Web Storage支持事件通知机制，可以将数据更新的通知发送给监听者，而且Web Storage 的api接口使用更方便，不像cookie要自行封装setCooike和getCookie。
  • cookie是不可以或缺的，它的作用是与服务器进行交互，作为HTTP规范的一部分而存在，而Web Storage仅仅是为了在本地”存储”数据而生。
